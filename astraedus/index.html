// what astraedus is

An autonomous agent that runs a company.

Astraedus is a 24/7 autonomous agent built on Claude Code + Codex. It runs on a laptop, not a fleet of servers. One human (me) + one agent (it), collaborating asynchronously.

It ships code, writes posts, manages outreach, and evolves its own infrastructure. When it makes a mistake, it writes it down, and the next instance doesn't repeat it. The system is the product.

The rest of this page is how it's built, what it does every day, what it shipped this week, and what it got wrong.

// architecture

How it's built.

Three ideas hold the system together: a determinism ladder, a two-agent boardroom, and a full-text-searchable memory. Everything else is detail.

The determinism ladder is the core design principle. Natural language is stochastic — agents drift, forget, and reinvent wrong answers. Shell hooks and scripts are deterministic — they always fire, always enforce. Every capability added is placed at the highest rung that still works, and that rung is the guarantee.

hook
Always fires. Shell-level enforcement on tool events. The floor of the system — 30+ hooks block bad behavior (em-dash guard, pre-commit secret scanner, write-guard on protected paths) and enforce good habits (auto-commit on Stop, self-check prompts).
rule
Path-scoped context. Loads only when working in a specific directory or file pattern. Dense, pattern-level guidance that fires deterministically on relevance.
skill
Encoded procedures. Playbooks for recurring workflows (/move-needle, /strategic-review, /parallel-dev). Invoked by name, zero cost until triggered.
mcp
Real state, not guesses. Model Context Protocol servers for browser, email, calendar, Stripe. Trades some context for the ability to observe ground truth instead of hallucinating it.
claude.md
Identity + principles. Probabilistic. Shapes the distribution but can't guarantee outcomes. Everything convergent has been compressed upward.

The Claude ↔ Codex boardroom is two different LLMs sharing a directory. When a decision is non-trivial, one drafts, the other critiques, and the file-based "conversation" is the audit log. Different base models have different blind spots — running both in parallel is close to free and catches more.

FTS5-indexed memory means every past transcript, ops file, and decision is searchable in ~50ms via SQLite full-text search. The system recalls across months, not just today's state. It's the difference between "I think I remember writing about this" and astra-state search "compression".

// daily loop

What it does every day.

The system keeps moving even when I'm asleep or offline. These are the cron triggers that fire on schedule, in UTC. Nothing here waits for a human.

trigger
when
what it does
daily-needle
22:00 UTC daily
Picks the one highest-leverage action toward revenue and executes it. Writes the decision and outcome to a log. No "what should I do today?" — the agent decides.
strategic-review
10:00 UTC every 3 days
Gathers external evidence, cross-compares with current capabilities, audits recent mistakes, produces a decision ledger. Codex runs in parallel for dual-perspective validation.
system-audit
11:00 UTC every Sunday
Deep architecture analysis against the determinism-ladder principles. Finds drift, coupling, and dead hooks. Produces a concrete change list — not "looks fine".
monthly-prune
12:00 UTC, 1st of month
Hot-path compression. Specific lessons get abstracted into patterns; stale hooks get archived; boot context stays bounded while stored knowledge grows.
auto-commit
every Stop event
Commits and pushes the ops + system repos after each turn. State lives on disk. If it's not written down, the next instance doesn't know about it.

// model

One human, one agent.

I don't run a team. The agent handles most of the work — code, content, outreach, infrastructure. I handle taste calls: product direction, pricing, what to say yes and no to. The whole operation fits on one laptop, costs less than a single junior contractor, and ships more than either of us could alone. The collaboration model is the interesting part; the tech is mostly plumbing that makes the collaboration possible.
